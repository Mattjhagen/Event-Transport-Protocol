# ETP v0.1 Specification (Draft)

## 1. Design Principles
The Event Transport Protocol is guided by six core principles:
1. **Events are objects, not files**: Events represent living state, not static snapshots.
2. **Event identity is immutable**: The EID is the permanent anchor for the event.
3. **Human-readable URLs are aliases**: Slugs and short-links are transient mappings to the EID.
4. **ICS is a compatibility layer only**: Legacy interop is a feature, not the foundation.
5. **Transport is protocol-agnostic**: EVT objects can be transported via HTTP, MQTT, or P2P.
6. **Synchronization is subscription-oriented**: State propagation relies on long-lived interests.

## 2. Architecture Layers
- **ETP**: The transport protocol layer handling routing and discovery.
- **EVT**: The canonical event object schema (JSON-based).
- **EID**: The Immutable Event Identity. MUST be globally unique and lexicographically sortable. The reference format is `evt_` followed by a **ULID** (e.g., `evt_01JX9P4M7G6XQ4A8N2Y`).
- **MIME**: `application/etp+json` for serialization.
- **URI**: `etp://{EID}` for global referencing.

## 3. Event Identity and Authority
### 3.1 Immutable Identity (EID)
The EID is the permanent anchor for the event. Once generated, an EID MUST NOT be reassigned to a different conceptual event.

### 3.2 Canonical Origin
Every EVT object SHOULD carry an `origin` field pointing to the authoritative ETP Node responsible for its state.
- **Authoritative Node**: The node that handles mutations and version increments.
- **Trust Model**: Clients verify origin authenticity via DNS (matching origin domain) or cryptographic signatures in the `auth` section.

## 4. Synchronization Semantics
ETP synchronization defines how event state propagates across the network.

### 4.1 Terminology
- **Snapshot**: A full representation of the current EVT object state.
- **Delta**: An incremental mutation (JSON Merge Patch or similar) representing a change between versions.
- **Stream**: An ordered transport channel (WebSockets, SSE) for delivering Snapshots or Deltas in real-time.
- **Subscription**: A persistent stateful relationship where a client requests updates for a specific EID.

### 4.2 Versioning and Mutation
- Every update to an object MUST increment the version (`v`) monotonically.
- The `updated_at` timestamp MUST reflect the moment the mutation was committed at the Origin.
- Clients SHOULD use the `rev` (revision hash) for quick delta-detection without parsing the full payload.

## 5. Protocol Capability Negotiation
Clients and Servers SHOULD negotiate protocol versions and transport capabilities via headers:
- `Accept: application/etp+json`
- `X-ETP-Version: 0.1`

## 6. Transport Independence
ETP is transport-agnostic. While this reference implementation uses HTTPS, the protocol is designed to operate over:
- **HTTPS**: For standard RESTful discovery and polling.
- **WebSockets / SSE**: For high-frequency streaming.
- **Native Channels**: IPC or app-to-app communication.

## 7. Routing Semantics
Routers MUST prioritize the EID for resolution. If an `alias` is used for discovery, the Router MUST provide the canonical EID in the response headers (`X-ETP-EID`).
