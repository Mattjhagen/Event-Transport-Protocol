# ETP v0.1 Specification (Draft)

## 1. Design Principles
The Event Transport Protocol is guided by six core principles:
1. **Events are objects, not files**: Events represent living state, not static snapshots.
2. **Event identity is immutable**: The EID is the permanent anchor for the event.
3. **Human-readable URLs are aliases**: Slugs and short-links are transient mappings to the EID.
4. **ICS is a compatibility layer only**: Legacy interop is a feature, not the foundation.
5. **Transport is protocol-agnostic**: EVT objects can be transported via HTTP, MQTT, or P2P.
6. **Synchronization is subscription-oriented**: State propagation relies on long-lived interests.

## 2. Architecture Layers
- **ETP**: The transport protocol layer handling routing and discovery.
- **EVT**: The canonical event object schema (JSON-based).
- **EID**: The Immutable Event Identity. MUST be globally unique and lexicographically sortable. The reference format is `evt_` followed by a **ULID** (e.g., `evt_01JX9P4M7G6XQ4A8N2Y`).
- **MIME**: `application/etp+json` for serialization.
- **URI**: `etp://{EID}` for global referencing.

## 3. Event Identity and Authority
### 3.1 Immutable Identity (EID)
The EID is the permanent anchor for the event. Once generated, an EID MUST NOT be reassigned to a different conceptual event.

### 3.2 Canonical Origin
Every EVT object SHOULD carry an `origin` field pointing to the authoritative ETP Node responsible for its state.
- **Authoritative Node**: The node that handles mutations and version increments.
- **Trust Model**: Clients verify origin authenticity via DNS (matching origin domain) or cryptographic signatures in the `auth` section.

## 4. Synchronization Semantics
ETP synchronization defines how event state propagates across the network.

### 4.1 Terminology
- **Snapshot**: A full representation of the current EVT object state.
- **Delta**: An incremental mutation (JSON Merge Patch or similar) representing a change between versions.
- **Stream**: An ordered transport channel (WebSockets, SSE) for delivering Snapshots or Deltas in real-time.
- **Subscription**: A persistent stateful relationship where a client requests updates for a specific EID.
- **ETP Node**: A server implementing the ETP API and maintaining authoritative event state.
- **ETP Subscriber**: A client maintaining a persistent connection for real-time updates.

### 4.2 Data Propagation Frames
ETP defines a set of standard synchronization frames used across all transport bindings:
- `snapshot.sync`: Contains a full versioned event object.
- `delta.sync`: Contains incremental changes and the new target version.
- `replay.sync`: Signals the start or end of a sequence of historical deltas.
- `heartbeat`: Liveness signal for persistent connections.
- `subscription.state`: Metadata describing the current sync health (active, recovering, stale).

### 4.3 Versioning and Replay
- Every update to an object MUST increment the version (`v`) monotonically.
- Reconnect handles should use `since=version` or `Last-Event-ID` headers.
- Nodes SHOULD maintain a delta history (sliding window) to fulfill replay requests without forcing full snapshots.

### 4.4 Temporal Lifecycle Semantics
ETP is intelligent about event-specific lifecycles. Defined states include:
- `scheduled`: The event is committed and expected to occur.
- `updated`: Generic mutation (content change).
- `delayed`: Start time is pushed forward, but the event remains scheduled.
- `rescheduled`: Significant time or date shift.
- `cancelled`: The event will no longer occur.
- `completed`: The event duration has passed.
- `archived`: Historical record, no further mutations expected.
- `expired`: Time-to-live (TTL) exceeded for ephemeral events.

### 4.5 Supersession Logic
When an event undergoes significant changes (e.g., location or platform shift), ETP uses versioned supersession:
- Mutations supersede all prior versions.
- The `supersedes` field MAY explicitly reference a prior EID or revision to resolve ambiguity in distributed caches.

### 4.6 Recurrence & Series
- `series_id`: An immutable identity for a recurring sequence.
- `occurrence_id`: A unique identity for a specific instance within that series.
- Mutations to a series SHOULD propagate to all occurrences unless they have "detached" overrides.

### 4.7 Timezone Authority
- All timestamps MUST be ISO 8601 UTC.
- The `timezone` field MUST be a canonical IANA Timezone (e.g., `America/New_York`).
- Nodes are responsible for DST calculations during mutation.

### 4.8 Temporal Trust Windows
Synchronization urgency increases as `start` approaches:
- **T-7 Days**: Regular polling/streaming (Normal).
- **T-24 Hours**: Higher heartbeat frequency (Critical).
- **T-1 Hour**: Direct PUSH/WebSocket priority (Real-time).

## 5. Trust & Authenticity
ETP provides cryptographic primitives to ensure event state remains authoritative and verifiable.

### 5.1 Node Identity
Nodes MUST express identity using a standard identity schema:
- `node_id`: Canonical domain or URI of the authority.
- `public_key`: The signing key used for mutations.
- `algorithm`: The cryptographic algorithm (e.g., `ed25519`).

### 5.2 Mutation Authenticity
Every Version (`v`) increment SHOULD be accompanied by a signature:
- Signatures MUST cover the `eid`, `v`, and the primary payload change.
- Mutations without verifiable signatures MAY be rejected by subscribers or marked as `unverifiable`.

### 5.3 Trust States
Clients MUST model the trustworthiness of incoming event data:
- `authoritative`: Verified direct connection to the Origin Node.
- `trusted`: Verified chain through known relay nodes.
- `bridged`: Probabilistic state from an interoperability bridge (ics/webcal).
- `downgraded`: Unverified legacy data.

### 5.4 Append-Only Lineage
EP objects maintain a "mutation lineage":
- Each version MUST conceptually link to its predecessor.
- Subscribers SHOULD verify the monotonic version progression.

## 6. Interoperability & Bridges
ETP recognizes that the legacy calendar ecosystem is vast and often static.

### 5.1 Fidelity Levels
Bridges MUST categorize their synchronization fidelity:
- `native`: Full ETP protocol support (streaming, replay, authority).
- `compatible`: Maintains live updates but MAY lose some semantic metadata (e.g., EID-aware CalDAV).
- `snapshot-only`: Static export. No synchronization guarantees (e.g., standard `.ics` download).

### 5.2 Semantic Downgrade
When ETP objects enter legacy systems, the protocol recognizes a "Semantic Downgrade":
- **Loss of Authority**: Legacy systems cannot verify EID origin signatures.
- **Loss of Synchronicity**: Streams collapse into periodic polls or manual refreshes.
- **Loss of Lifecycle Transitions**: Complex states (delayed, rescheduled) are often flattened to generic "Updated".

### 5.3 Bridge Node Advertisement
Bridge nodes SHOULD advertise their translation capabilities via `GET /api/bridges`:
- Supported formats (ICS, WebCal, CalDAV).
- Fidelity rating.
- Known degradations (e.g., "Timezone info may be simplified").

## 6. Transport Bindings
ETP remains transport-neutral via specific binding definitions:

### 5.1 SSE Binding
- **Endpoint**: `/api/e/:id/stream`
- **Framing**: Standard text/event-stream using `id:` for version and `data:` for JSON-encoded frames.
- **Replay**: Uses `Last-Event-ID` header or `since` query parameter.

### 5.2 WebSocket Binding
- **Endpoint**: `/api/etp-ws`
- **Framing**: JSON-wrapped messages.
- **Action**: `etp.subscribe` initiate subscription.
- **Multiplexing**: A single WS connection SHOULD support multiple EID subscriptions.

## 6. Capability Negotiation and Discovery
### 6.1 Capability Endpoint
Nodes SHOULD expose a discovery endpoint at `GET /api/capabilities` returning:
- Supported protocol versions.
- Supported transport bindings.
- Feature flags (replay, compression).
- Constraints (max_replay_depth, heartbeat_interval).

### 6.2 Client Negotiation
Clients express intent through specialized headers or initialization frames:
- Transport preference.
- Snapshot requirements.
- Mutation idempotency tokens.

## 6. Transport Independence
ETP is transport-agnostic. While this reference implementation uses HTTPS, the protocol is designed to operate over:
- **HTTPS**: For standard RESTful discovery and polling.
- **WebSockets / SSE**: For high-frequency streaming.
- **Native Channels**: IPC or app-to-app communication.

## 7. Routing Semantics
Routers MUST prioritize the EID for resolution. If an `alias` is used for discovery, the Router MUST provide the canonical EID in the response headers (`X-ETP-EID`).
